# Technical Spec — Client-Side Reminders v1

## 1. Overview

Add local push notifications for habit reminders. The Service Worker displays notifications at each habit's scheduled time, on due days only. No server-side components.

## 2. Current State

- **API:** `Habit.reminder { enabled: Boolean, time: String }` — time is HH:MM in UTC
- **Web:** HabitForm has toggle + time picker, stores via `createHabit`/`updateHabit` mutations
- **Service Worker:** Auto-generated by `vite-plugin-pwa` (Workbox) for caching only
- **Gap:** Reminder data is stored but never triggers any notification

## 3. Implementation Plan

### Phase 1: Service Worker with Notification Support

Switch `vite-plugin-pwa` from `generateSW` (default) to `injectManifest` mode so we can add custom notification logic to the Service Worker.

**Files:**
- `web/vite.config.ts` — switch to `injectManifest`
- `web/src/sw.ts` — new custom Service Worker source

**Service Worker responsibilities:**
1. Listen for `message` events from the app with type `SCHEDULE_REMINDERS`
2. Store scheduled reminders in a `Map<habitId, timeoutId>`
3. Use `setTimeout` to fire `self.registration.showNotification()` at the right time
4. Listen for `notificationclick` to open the app at `/streaks/{habitId}`
5. Listen for `CANCEL_REMINDER` and `CANCEL_ALL_REMINDERS` messages

**Service Worker message protocol:**
```typescript
// App → SW
{ type: 'SCHEDULE_REMINDERS', reminders: Array<{ habitId: string, habitName: string, delayMs: number }> }
{ type: 'CANCEL_REMINDER', habitId: string }
{ type: 'CANCEL_ALL_REMINDERS' }

// SW behavior
- On SCHEDULE_REMINDERS: clear existing timeouts, set new ones
- On timeout: call showNotification("Ready to [habitName]?", { data: { habitId }, ... })
- On notificationclick: open /streaks/{habitId} via clients.openWindow()
```

### Phase 2: Reminder Scheduler Hook

**File:** `web/src/hooks/useReminderScheduler.ts`

A hook that runs on app load and whenever habits change. It:
1. Reads all habits from Apollo cache
2. Filters to habits with `reminder.enabled = true`
3. For each, checks if today is a due day (based on frequency/dueDays)
4. Checks if already completed today (skip notification)
5. Converts `reminder.time` (UTC HH:MM) to local time
6. Computes `delayMs` — milliseconds until that local time today
7. If `delayMs < 0` (time already passed today), skips
8. Posts `SCHEDULE_REMINDERS` message to the Service Worker

**Scheduling logic:**
```typescript
function computeDelay(utcTime: string): number {
  const [h, m] = utcTime.split(':').map(Number);
  const now = new Date();
  const target = new Date();
  target.setUTCHours(h, m, 0, 0);
  return target.getTime() - now.getTime();
}
```

**Where to mount:** In `App.tsx`, inside `AuthProvider` so it has access to the authenticated user's habits.

### Phase 3: Permission Request in HabitForm

**File:** `web/src/components/habits/HabitForm.tsx`

When user toggles "Enable reminder" ON:
1. Check `Notification.permission`
2. If `'default'` → call `Notification.requestPermission()`
3. If `'denied'` → show inline warning: "Notifications are blocked. Enable them in your browser settings."
4. If `'granted'` → proceed normally
5. If permission request returns `'denied'` → uncheck the toggle, show warning

**New component:** `web/src/components/common/PermissionWarning.tsx` — small inline warning banner.

### Phase 4: Reschedule on Habit Changes

After any `createHabit`, `updateHabit`, or `deleteHabit` mutation:
- The Apollo cache updates
- `useReminderScheduler` reacts to the change and re-posts `SCHEDULE_REMINDERS`
- For delete: post `CANCEL_REMINDER` for that habit

The hook watches `habits` data from the Apollo query, so cache updates trigger rescheduling automatically.

## 5. Notification UX

- **Title:** "Indago"
- **Body:** "Ready to [habit name]?"
- **Icon:** `/pwa-192x192.png`
- **Tag:** `reminder-{habitId}` (prevents duplicates)
- **Click action:** Open `/streaks/{habitId}`
- **Require interaction:** false (auto-dismiss after OS default timeout)

## 6. Edge Cases

| Case | Behavior |
|------|----------|
| App not open | SW may be killed — notifications won't fire. Known v1 limitation. |
| Multiple habits same time | Separate notification per habit |
| Habit completed before reminder time | Skip notification (checked at schedule time and recheck at fire time) |
| User changes timezone (travel) | Recalculated on next app load since we use local `Date` |
| Reminder time in the past today | Skip until tomorrow (tomorrow handled on next app load) |
| Browser doesn't support notifications | Toggle still works (data saved), no notification fires, graceful no-op |

## 8. Open Questions

| # | Question | Resolution |
|---|----------|------------|
| 1 | Should `injectManifest` break existing Workbox caching? | No — we import the Workbox precache manifest in our custom SW and add notification logic on top. |
| 2 | Does `setTimeout` survive in a dormant SW? | No — but the SW stays alive while there are active timeouts. Chrome keeps it alive for up to 5 minutes of inactivity. We reschedule on app load to compensate. |
